# ArchGuard Methodology — Source of Truth

The single source of truth for the Architectural Guard Methodology. This is the immutable constitution for AI-native development. All skills (`archguard-discover`, `archguard-clarify`, `archguard-plan`, `archguard-implement`, `archguard-analyze`) derive their behavior from this document.

---

## Part 1: The 7 Failure Modes (The Axioms)

These are universal. They apply to every project, in every domain. They define *why* agent shortcuts are dangerous and *what tests cannot see*.

| # | Failure Mode | The Agent Shortcut | The Invariant | The CS Principle |
|:---|:---|:---|:---|:---|
| **FM1** | **Trusts Input** | Passes raw external data into internal state without validation. | **I.1 Data Boundary** | Complete Mediation (Saltzer & Schroeder, 1975) |
| **FM2** | **Inverts Dependencies** | Imports concrete infrastructure into abstract business logic. | **I.2 Dependency Direction** | Dependency Inversion (Martin, 1996) / Information Hiding (Parnas, 1972) |
| **FM3** | **Leaks State** | Dumps raw internal state (PII, secrets) into logs, responses, or telemetry. | **I.3 Leakage Gate** | Lattice Model of Information Flow (Denning, 1976) |
| **FM4** | **Ignores Failure** | Makes external calls without timeouts; writes empty catch blocks; skips circuit breakers. | **I.4 Resilience** | Fail-Fast (Gray, 1985) / Circuit Breaker (Nygard, 2007) |
| **FM5** | **Skips Cleanup** | Opens connections/streams without guaranteed teardown. | **I.5 Resource Lifecycle** | RAII (Stroustrup) / Deterministic Destruction |
| **FM6** | **Writes Partial State** | Executes multiple mutations sequentially without atomicity. | **I.6 State Mutation** | ACID (Gray & Reuter, 1992) / Linearizability |
| **FM7** | **Corrupts In-Flight State** | Allows transient/in-flight data to violate invariants that at-rest data enforces. Skips re-validation between pipeline steps or after serialization. | **I.7 Invariant Preservation** | Hoare (1969) Axiomatic Semantics / Design by Contract (Meyer, 1986) |

### Formal Completeness Proof

von Neumann (1945) established that every digital computer is a stored-program machine with exactly five components: **Input**, **Output**, **Memory**, **Control**, and **Arithmetic Logic**. Any statically analyzable program can be reduced to this architecture:

```
State Machine = f( Input, Current State, Resources ) → Output + New State
```

Architectural violations always manifest as a failure in one of the von Neumann surfaces — no program can fail in a way that escapes these seven categories because no program can *express* a failure outside them:

| FSM Component | von Neumann Unit | Requirement | Failure Mode |
|:---|:---|:---|:---|
| **Input** | Input | Validated before entering state machine | FM1 (Trusts Input) |
| **Internal Logic** | Control + ALU | Correct dependency references | FM2 (Inverts Dependencies) |
| **Output** | Output | No exposure of internal state | FM3 (Leaks State) |
| **Transition** | Control | Graceful error handling + timeouts | FM4 (Ignores Failure) |
| **Resources** | Memory (working) | Released after transition completes | FM5 (Skips Cleanup) |
| **State Persistence** | Memory (permanent) | Atomic updates | FM6 (Writes Partial State) |
| **Transient State** | Memory (in-transit) | Invariants preserved through all data flows | FM7 (Corrupts In-Flight State) |

The 7 FMs are **exhaustive**. If you can reduce a program to von Neumann architecture (as all stored-program computers can be), it has exactly these 7 failure surfaces. The 15 invariant bans are the concrete syntactic manifestations — each ban maps to one of the 7 surfaces, and together they cover every path by which an agent shortcut can bypass test coverage.

## Part 2: The 4 Structural Maps

To find where the FMs will manifest, scan the codebase for these 4 structural maps. These are generated by `archguard-discover` via pattern matching — no domain knowledge required.

### Map A: Trust Boundaries (data enters the system)

| Category | Patterns to find | FM |
|:---|:---|:---|
| HTTP request objects | `req.body`, `req.params`, `request.json()`, `ctx.request.body` | FM1 |
| WebSocket / realtime events | `.on('message')`, `.onmessage`, `.subscribe(` | FM1 |
| CLI / process arguments | `process.argv`, `process.env` reads | FM1 |
| File system reads | `fs.readFile`, `fs.createReadStream` | FM1 |
| Job queue consumers | Worker `process()` callbacks | FM1 |
| Browser / platform events | `addEventListener(`, `postMessage` handlers | FM1 |

### Map B: Dependency Graph (import relationships)

1. Identify layer boundaries from directory structure
2. Classify each directory: Domain/Core, Infrastructure/Adapters, Entrypoints/Transport, Features/Modules, Apps/UI, Scripts
3. Determine allowed dependency directions (inner → outer is banned)
4. Flag all cross-boundary imports

| Violation | FM |
|:---|:---|
| Domain layer importing from Infrastructure | FM2 |
| Server-side code importing browser-only libraries | FM2 |
| Entrypoints importing from other entrypoints | FM2 |

### Map C: Output Surfaces (data leaves the system)

| Category | Patterns to find | FM |
|:---|:---|:---|
| HTTP responses | `.json()`, `.send()`, `return` in route handlers | FM3 |
| Logging calls | `.info(`, `.error(`, `.warn(`, console methods | FM3 |
| Telemetry / tracing | `span.setAttribute(`, `span.addEvent(`, `setTag(` | FM3 |
| WebSocket outbound | `.send(`, `.emit(`, `.broadcast(` | FM3 |
| Error propagation | Error constructors with metadata, `throw new` with details | FM3 |

### Map D: Resources + Mutations (connections opened, state written)

| Category | Patterns to find | FM |
|:---|:---|:---|
| Database mutations | For EACH DB driver found: `.insert(`, `.update(`, `.delete(`, `.execute(`, `.run(` | FM6 |
| Stream/file handles | `createWriteStream`, `createReadStream` | FM5 |
| Crypto operations | `createCipheriv`, `generateKeyPair` | FM5 |
| Realtime connections | For EACH realtime library: connect/open/join methods | FM5 |
| Process termination | `process.exit(`, runtime-specific exit calls | FM5 |
| External network calls | `fetch(`, framework-specific HTTP clients | FM4 |

**FM6 detection (multi-mutation):** Flag any function body with 3+ consecutive write calls without a transaction wrapper.

## Part 3: Tech Context — The 5 Escape Hatch Questions

For every dependency in `package.json`, apply these 5 universal questions. The questions are fixed. The answers come from the dependency's actual API surface.

| # | Question | If "Yes" | FM |
|:---|:---|:---|:---|
| **Q1** | **Raw escape hatch?** Can raw/untrusted strings reach an engine/interpreter? | Ban raw string interpolation; force parameterized API | FM1 |
| **Q2** | **Unbounded default?** Can it be created without explicit limits? | Ban instantiation without maxSteps/maxTokens/maxRetries/maxConnections | FM4 |
| **Q3** | **Trust boundary?** Is output treated as safe without sanitization? | Ban passing output directly to storage/responses without sanitization | FM1 / FM3 |
| **Q4** | **Client/server split?** Are there platform-specific variants? | Ban platform-specific imports in wrong context | FM2 |
| **Q5** | **Missing timeout?** Can it be called without a deadline/abort? | Ban calls without timeout/signal/AbortController | FM4 |

Dependencies that pass all 5 questions need no bans. That is fine — most dependencies are not risks.

## Part 4: The 15 Invariant Bans (Universal, Non-Negotiable)

These are the concrete syntactic bans that enforce the 7 Failure Modes. They are auto-generated by `archguard-discover` and included by `archguard-plan`. No user confirmation is needed — they are axioms.

| # | Ban | FM | ESLint Rule / Selector | Activation |
|:---|:---|:---|:---|:---|
| 1 | Raw `JSON.parse()` | FM1 | `CallExpression[callee.object.name='JSON'][callee.property.name='parse']` | Always |
| 2 | `any` type + `as any` | FM4 | `@typescript-eslint/no-explicit-any: error` | Always |
| 3 | `@ts-ignore` / `@ts-nocheck` | FM4 | `@typescript-eslint/ban-ts-comment: error` | Always |
| 4 | `console.log` (and info/warn/debug) | FM3 | `CallExpression[callee.object.name='console'][callee.property.name=/^(log\|info\|warn\|debug)$/]` | Always |
| 5 | `fetch()` without `AbortSignal` | FM4 | `CallExpression[callee.property.name='fetch']:not([arguments.1.properties[name='signal']])` | If `fetch()` used |
| 6 | Empty `catch` blocks | FM4 | `CatchClause[body.body.length=0]` | Always |
| 7 | `process.exit()` / `Bun.exit()` | FM5 | `CallExpression[callee.object.name='process'][callee.property.name='exit']` | Node.js / Bun |
| 8 | `while(true)` / `for(;;)` + `setInterval` | FM4/FM5 | `WhileStatement[test.value=true]`, `ForStatement[init=null][test=null][update=null]`, `CallExpression[callee.name='setInterval']` | If found |
| 9 | `process.env` outside config | FM1 | `MemberExpression[object.object.name='process'][object.property.name='env']` (excl. config dir) | Node.js |
| 10 | `Date.now()` / `new Date()` in business logic | FM7 | `NewExpression[callee.name='Date']`, `CallExpression[callee.object.name='Date'][callee.property.name='now']` | Always |
| 11 | Floating promises (un-awaited) | FM5/FM4 | `@typescript-eslint/no-floating-promises: error` | Always |
| 12 | `export *` (barrel exports) | FM3 | `ExportAllDeclaration` | Always |
| 13 | Mutable module-level state (`let`/`var` at module scope) | FM6 | `VariableDeclaration[declarations.kind!='const']:not(:has(ancestor::BlockStatement))` | Always |
| 14 | `Math.random()` / `crypto.randomUUID()` in business logic | FM7 | `CallExpression[callee.object.name='Math'][callee.property.name='random']`, `CallExpression[callee.object.name='crypto'][callee.property.name='randomUUID']` | Always |
| 15 | Untyped `catch` variable (TypeScript) | FM4 | `tsconfig.json: useUnknownInCatchVariables: true` (enforced by `"strict": true`) | Always |
| 16 | Adapter `new` outside composition root (e.g. `new PrismaUserRepository()` in route files) | FM2 | `NewExpression[callee.name=/^(AdapterClass1\|AdapterClass2\|...)$/]` (excl. main.ts / app.ts / container.ts) | Always |

### Per-primitive external client bans (from Tech Context Q1-Q5)

These are NOT in the invariant table because the specific selectors depend on which packages are installed. `discover` generates them via the 5 Questions. Examples (not exhaustive):

- Neo4j: `session.run()` with template expressions → FM1
- Prisma: `$queryRaw`, `$executeRaw` → FM1
- Supabase: `.rpc()` bypassing RLS → FM1
- Mastra: `new Agent()` without `maxSteps` → FM4
- Vercel AI SDK: `generateText()` returned raw → FM1
- LiveKit: `livekit-client` in server paths → FM2

### Database-Layer Bans (Universal, Activation-Conditional)

These 8 bans extend the 15 invariant bans to the persistence layer. They are universal — every RDBMS-backed application has these failure surfaces — but activation is conditional on the presence of a database adapter in the project (same tier as Ban 5 "`fetch()` without `AbortSignal`": always true in principle, only enforced when the primitive exists in the codebase).

The von Neumann mapping for DB concerns: persistence is the permanent Memory unit. Every write to permanent memory creates a state transition, and every read from permanent memory is an input to the in-memory state machine. The same 7 FMs apply — the DB is just another surface where they manifest.

| # | Ban | FM | von Neumann Surface | CS Principle | Detection | Activation |
|:---|:---|:---|:---|:---|:---|:---|
| 17 | Unbounded `text()` / `TEXT` columns for finite-domain fields | FM1 | Input (Memory — permanent) | **Buffer Overflow (One, 1996).** An unbounded column tells the storage engine "allocate whatever arrives." At the page-buffer level, a single 10MB string poisons the L1/L2 cache lines for every subsequent read of that table. `varchar(N)` is Complete Mediation (Saltzer & Schroeder, 1975) at the disk boundary — the column width is the upper bound on what enters persistent state. | Schema scan: `text()` or `TEXT` columns whose domain is known-finite (identifiers, emails, phone numbers, status codes, names). In Drizzle: `text()` without `.max()`. In SQLAlchemy: `Text()` without `length`. | If ORM/schema present |
| 18 | Naked string enums (categorical fields without `pgEnum` / `CHECK`) | FM1 + FM7 | Input + Transient State | **Design by Contract (Meyer, 1986).** The CHECK constraint is the invariant clause of the write contract. If the application's in-flight validation is bypassed (FM7), the DB engine still rejects the malformed row. This is the Liskov Substitution Principle at the persistence layer — any value written must satisfy the column's contract, and the DB enforces it regardless of which code path originated the write. Without the constraint, a single bug in application validation permanently corrupts the data set. | Schema scan: `text()` columns whose values come from a closed set (< 12 known values). In PostgreSQL: no `pgEnum` / `CHECK (col IN (...))` on the column. | If ORM/schema present |
| 19 | Unindexed foreign key columns (`.references()` without index) | FM4 | Transition | **B-tree Indexing (Bayer & McCreight, 1972).** A foreign key join on an unindexed column forces the query planner into a sequential scan — O(n) per lookup, O(n × m) for the join. At production volume, this saturates the CPU and triggers statement timeouts (FM4). The CS invariant: every edge in the relational graph must have a lookup structure. The index is not a performance optimization — it is the structural guarantee that the edge is traversable in bounded time. | Schema scan: `.references()` / `FOREIGN KEY` declaration without corresponding `index()` / `CREATE INDEX`. | If ORM/schema present |
| 20 | Concurrent queries inside a single transaction (`Promise.all` / `asyncio.gather` on `tx.*`) | FM6 | State Persistence | **ACID Isolation (Gray & Reuter, 1992).** A database transaction maps to exactly one synchronous network socket. Pipelining concurrent queries down a single connection deserializes the wire protocol — the driver cannot distinguish which result belongs to which query, and the DB may interleave partial results from concurrent statements. Two-Phase Locking (Eswaran et al., 1976) assumes serial statement execution per connection. | AST scan: `Promise.all([...tx...])`, `asyncio.gather(*[...tx...])`, `sync.WaitGroup` with `tx.Query` inside goroutines. | If DB adapter present |
| 21 | Database queries inside loops (N+1) | FM4 | Resources | **Amdahl's Law (1967).** Each DB query incurs network round-trip latency RTT. Inside a loop, total time = n × RTT — the serial network I/O dominates regardless of how fast the DB executes each query. At n = 100 items and RTT = 2ms, a 200ms operation becomes 20s. This is not a style concern — it is unbounded resource consumption that exhausts connection pools and triggers cascading timeouts (FM4). The fix is always structural: batch criteria (WHERE id IN (...)) or a JOIN. | AST scan: DB client call (`.select()`, `.insert()`, `.execute()`, `.query()`) inside `for` / `.forEach` / `.map` / list comprehension body. | If DB adapter present |
| 22 | Multi-statement writes without transaction wrapper | FM6 | State Persistence | **ACID Atomicity (Gray & Reuter, 1992).** Two or more `.insert`/`.update`/`.delete` calls in the same function without `db.transaction()` means partial writes are visible to concurrent readers. The window between statement 1 committing and statement 2 executing is a dirty read surface — another transaction sees the intermediate state and makes decisions on it. This is a violation of the C in ACID: concurrent transactions observe inconsistent states. | AST scan: function body with 2+ write calls (`.insert`, `.update`, `.delete`, `.execute`, `.run`, `.save`) and no wrapping `transaction()` / `tx()` call. | If DB adapter present |
| 23 | Connection pool without drain path | FM5 | Resources | **RAII (Stroustrup).** `new Pool()` acquires file descriptors from the OS process limit (ulimit). Without a corresponding `pool.end()` reachable from the SIGTERM/SIGINT handler, those descriptors leak on every restart. In containerized environments, leaked descriptors persist until the kernel reclaims them (TCP TIME_WAIT: 60s), causing "too many open files" on the next deployment. The CS invariant: resource acquisition and release must be collocated in the same lexical scope or lifecycle hook. | AST scan: `new Pool()` / `createPool()` with no exported `close()` / `end()` / `destroy()` method, or no integration with the project's shutdown handler. | If Node.js + PG / connection pool present |
| 24 | Unbounded audit log payloads (raw entity `jsonb` in audit tables) | FM3 + FM1 | Output + Input | **Lattice Model of Information Flow (Denning, 1976).** Audit tables are an output surface — they exist for operational visibility. Storing raw entity objects (including PII) in audit `jsonb` columns is a lattice violation: high-classification data (PII) flows into a low-classification sink (observability). The audit record should contain *what happened*, not *the data it happened to*. This is the DB equivalent of Ban 4 (`console.log`): the output surface must be bounded and PII-free. | Schema scan: `jsonb` / `JSONB` audit/event/log columns that accept raw entity objects (not structured event types). | If audit/event tables present |

### DB-Layer Whitelist Stubs

Each DB-layer ban has a whitelist pattern — the single file or migration where the banned pattern is permitted:

| Ban # | Primitive | Whitelist Location | Safety Mechanism |
|:---|:---|:---|:---|
| 17 | `text()` / `TEXT` | `<adapter-layer>/db/schema.ts` (DDL definition file) | Every `text()` must declare `.max(N)` or be justified by a comment naming the unbounded domain |
| 18 | `text()` for enums | Same DDL file | Column must have `pgEnum()` or `check()` constraint in the same migration |
| 19 | `.references()` without index | Same DDL file | Every FK column must have `.$defaultFn()` or separate `index()` declaration adjacent to the column definition |
| 20 | Concurrent tx queries | `<adapter-layer>/db/transaction.ts` | Single helper that enforces sequential `await` per statement |
| 21 | Queries in loops | Nowhere — no whitelist | N+1 is never correct. Always batch or join |
| 22 | Multi-write without tx | `<adapter-layer>/db/transaction.ts` | Transaction helper wraps all writes in `db.transaction()` |
| 23 | Pool without drain | `<adapter-layer>/db/client.ts` | Pool creation file exports `close()`; `<config-layer>/shutdown.ts` calls it on SIGTERM |
| 24 | Audit `jsonb` of entities | `<adapter-layer>/db/audit-schema.ts` | Audit columns store `event_type`, `entity_type`, `entity_id` (FK), `diff` (field-level), `timestamp` — never raw entity JSON |

## Part 5: The Whitelist Rule

**Every ban must have exactly ONE whitelist file** where the banned primitive is allowed.

| Primitive | Whitelist File (example paths) | Safety Wrapper |
|:---|:---|:---|
| `fetch()` | `<adapter-layer>/http/safe-fetch.ts` | Timeout (AbortSignal) + Zod validation |
| `JSON.parse` | `<core-layer>/safe-parse.ts` | JSON.parse inside Zod.safeParse |
| `process.env` | `<config-layer>/env-schema.ts` | Centralized env parsing + validation |
| `console.log` | `<project-logger-file>` | Structured logger with trace IDs |
| `process.exit` | `<core-layer>/shutdown.ts` | Graceful SIGTERM/SIGINT handler |
| `new AdapterClass()` outside main.ts | `main.ts` / `app.ts` / `container.ts` | Composition root — the single file where all adapters are instantiated and injected |
| `Date.now()` / `new Date()` | `<core-layer>/time-service.ts` | Injectable time abstraction |
| `Math.random()` / `crypto.randomUUID()` | `<core-layer>/id-generator.ts` | Injectable ID generation + controlled randomness |
| DB driver queries | `<adapter-layer>/<db>/parameterized-query.ts` | Parameterized queries only |

The whitelist stub enforces what the ban requires — timeout + signal + validation. Implementation details (retry, backoff, idempotency keys) are operational decisions per-endpoint, not part of the architectural contract.
